# Deploying Ripple Transit (Render + Turso)

Ripple runs as a single Node web service that serves the API and the built
client on one port, backed by a Turso (libSQL) database. Routing/search stay on
OneMap/LTA/HERE via server-side keys.

---

## 1. Create the database (Turso)

```bash
# Install + sign in (one-time)
brew install tursodatabase/tap/turso   # or: curl -sSfL https://get.tur.so/install.sh | bash
turso auth signup

# Create the DB (Singapore region keeps latency low)
turso db create ripple-transit --location sin

# Grab the two values you'll paste into Render:
turso db show ripple-transit --url          # ‚Üí DATABASE_URL  (libsql://‚Ä¶)
turso db tokens create ripple-transit       # ‚Üí DATABASE_AUTH_TOKEN
```

(Optional) Apply the schema from your machine to sanity-check before deploying:

```bash
DATABASE_URL="libsql://‚Ä¶" DATABASE_AUTH_TOKEN="‚Ä¶" npm run db:migrate
DATABASE_URL="libsql://‚Ä¶" DATABASE_AUTH_TOKEN="‚Ä¶" NODE_ENV=production npm run db:seed
```

The Render start command also runs migrate + seed automatically on every deploy,
so this step is just a check.

---

## 2. Create the web service (Render)

**Option A ‚Äî Blueprint (recommended).** The repo ships a `render.yaml`.

1. Render Dashboard ‚Üí **New ‚Üí Blueprint** ‚Üí connect `Wongjon1994/Ripple-Transit`.
2. Render detects `render.yaml` and proposes the `ripple-transit` web service.
3. Apply ‚Äî it provisions the service and auto-generates `SESSION_SECRET`.
4. Open the service ‚Üí **Environment** and fill the `sync: false` secrets (below).
5. **Manual Deploy** (or push to `main`) to build with the secrets in place.

**Option B ‚Äî manual web service.** New ‚Üí Web Service ‚Üí this repo, then set:

| Field | Value |
|-------|-------|
| Runtime | Node |
| Build command | `npm ci --include=dev && npm run build` |
| Start command | `npm run start:prod` |
| Health check path | `/api/health` |

## 3. Environment variables (Render ‚Üí Environment)

Auto-set by the blueprint:

| Var | Value |
|-----|-------|
| `NODE_ENV` | `production` |
| `SESSION_SECRET` | auto-generated by Render |
| `HERE_MONTHLY_CAP` | `29950` |
| `PORT` | provided by Render |

You paste these (from Turso + your API providers):

| Var | Where it comes from |
|-----|---------------------|
| `DATABASE_URL` | `turso db show ‚Ä¶ --url` |
| `DATABASE_AUTH_TOKEN` | `turso db tokens create ‚Ä¶` |
| `ONEMAP_TOKEN` | OneMap 3-day JWT (same as local `.env`) |
| `ONEMAP_EMAIL` / `ONEMAP_PASSWORD` | optional ‚Äî enables token auto-refresh |
| `LTA_ACCOUNT_KEY` | LTA DataMall |
| `HERE_API_KEY` | HERE |

> Cookies are `Secure` + same-origin in production and the server trusts the
> Render proxy, so sign-in works over HTTPS with no extra config.

---

## 4. Deploy & verify

1. Trigger a deploy (push to `main` with `autoDeploy`, or Manual Deploy).
2. Watch logs: you should see migrations apply, MRT lines seed
   (`Skipping dev user (NODE_ENV=production)`), then
   `‚ñ≤ Ripple Transit on http://localhost:<PORT> (production)`.
3. Visit the Render URL:
   - `GET /api/health` ‚Üí `{"ok":true,"env":"production"}`
   - Register an account (there is **no** default admin in production),
     search a route, confirm live bus feasibility.

---

## Notes

- **First user is not an admin.** The seed no longer creates `dev@ripple.transit`
  in production. Register normally; to grant admin, update the `users.role` row
  in Turso (`turso db shell ripple-transit`).
- **OneMap token lifespan.** The provided token is a 3-day JWT. Set
  `ONEMAP_EMAIL`/`ONEMAP_PASSWORD` to let the server auto-refresh it; otherwise
  rotate `ONEMAP_TOKEN` when it expires.
- **Free plan** spins down on idle; the first request after a cold start is slow.
- **Custom domain / CORS.** Same-origin needs no CORS config. If you later split
  the client onto another origin, set `CLIENT_ORIGIN` to that URL.
